# Anthropic发布Computer Use最佳实践：想让AI点得更准，请先把截图缩小

**来源**: AGI Hunt  
**日期**: 2026-05-14

---

## 核心要点

Anthropic 发布了 Computer Use 开发者最佳实践，涵盖截图处理、模型选择、上下文管理、提示注入防护等关键主题。

---

## 01 缩小截图

**最重要的一条建议：发送给 API 前先把截图降采样。**

**原因：** Claude API 内部有图片分辨率上限：
- Claude 4.6 系列：长边 1568 像素、总像素 1.15MP
- Opus 4.7：长边 2576 像素、总像素 3.75MP

超过限制后 API 会自动缩小，但返回的坐标是基于缩小后的尺寸，而客户端还在用原始分辨率执行点击 → **坐标错位**。

**推荐起始分辨率：** 1280×720（约 80% 像素预算）；Opus 4.7 可直接上 1080p。

**注意：** 坐标要缩放回去 — 模型返回的是缩小尺寸的坐标，执行时需乘以缩放系数还原到屏幕真实坐标。

**macOS 坑点：** macOS 截图默认 2 倍 DPI，1440p 屏幕截出来实际是 2880p，不处理的话点击准确率会崩。

---

## 02 先说再看

构造消息时，**文字指令要放在图片前面**。

推荐顺序：先文字「点击提交按钮」，再看截图。这样模型知道该找什么。

原理类似：先告诉人找哪份合同，再把文件堆在面前，效率更高。

---

## 03 谁点得准

不同模型各有所长：

| 模型 | 特点 |
|------|------|
| **Sonnet 4.6** | 点击精度最高，空间定位准，近距离失误少，对图片压缩容忍度更好。适合机械执行类任务 |
| **Opus 4.7** | 推理能力更强，点击精度已追平 Sonnet。分辨率预算更大（3.75MP vs 1.15MP），需要压缩幅度更小。既需推理又需精准点击的最优解 |
| **Haiku 4.5** | 延迟优先，适合对速度要求极高的场景 |

**高级模式：** 用推理能力强的模型做「指挥官」负责规划，让 Sonnet/Haiku 去执行具体点击操作。

---

## 04 目标太小怎么办

4K 屏幕（3840×2160）压缩到 720p 后，16 像素的复选框只剩约 5 个像素。

**解法：**
1. **开 zoom**：Claude 4.6/4.7 支持 `enable_zoom: True`，让模型先放大区域再看清楚再点击
2. **把目标放大**：降低系统 DPI、放大浏览器缩放、调整 UI 比例
3. **用键盘替代**：Tab 导航或快捷键比鼠标点击更靠谱

## 05 思考强度

| 模型 | 推荐思考力度 |
|------|-------------|
| Opus 4.7 | high（准确率接近 max，但 token 消耗只有一半） |
| Claude 4.6 系列 | medium（任务成功率已接近峰值，再往上加就是浪费） |

**不推荐在 Computer Use 中使用 max effort** — 比 high 没有任何准确率提升，只会增加 token 成本。

## 06 提示注入防护（三层城墙）

1. **训练层面**：模型在训练时就学会识别和拒绝恶意指令
2. **实时分类器**：并行扫描文本、图片中隐藏的指令和伪造 UI 元素，零额外延迟、零额外成本（使用 `computer_20251124` 工具类型时自动运行）
3. **持续红队测试**：安全研究人员不断试探防线

**最有效的防御是 human-in-the-loop**：在执行不可逆操作（提交表单、付款、发消息、改数据）前，让 agent 暂停请用户确认。

## 07 长对话处理（三层上下文管理）

每次截图消耗 1000-1800 个 token，200K 上下文窗口不到 100 张截图就满了。

**三层策略：**

### 第一层：缓存断点
API 支持最多 4 个缓存断点。推荐：
- 1 个放在系统提示和工具定义上（每次都一样）
- 另外 3 个放在最新的 tool_result 上，每轮清除旧标记、放置新标记

### 第二层：滚动缓冲
只保留最近 N 张截图的完整数据，更早的截图替换成 `[Image omitted]`。

默认保留最近 3 张（`keep_n=3`），每累积 25 张才做一次清理（`interval=25`）。批量替换能让前缀在连续多轮里保持字节一致，缓存命中率才能上去。

### 第三层：LLM 压缩
用模型总结对话历史，保留 8 类信息：
- 用户的完整指令（逐字保留所有「必须」「不要」「始终」这类约束）
- 任务模板、约束规则
- 已执行的操作
- 出错及修复记录
- 进度追踪、当前状态
- 下一步要做什么

**服务端自动版本：** 在 API 请求里加 `context_management` 参数和 `compact-2026-01-12` beta 标识，触发自动压缩。

## 08 教学模式（最有趣的部分）

**新思路：别告诉 Claude 怎么做，直接「示范」给它看。**

做法：用户先手动执行一遍任务，系统录制每一步操作（包括点击坐标、输入内容、页面导航），每步配一张截图，用蓝色圆圈标注点击位置。

Claude 收到的是完整操作示范，理解「要做什么」，然后在当前真实环境里执行相同操作。如果 UI 布局变了、按钮移了位置，Claude 会根据示范理解目标，在新界面上找到对应元素。

**三种回放模式：**
- **严格模式**：完全按步骤执行，UI 变化太大就停下来报告。适合合规敏感场景
- **自适应模式**（推荐）：以示范为参考但灵活调整，能应对轻微布局变化、按钮改名、菜单重排
- **目标导向模式**：只关注最终结果，录制的步骤只是参考。适合 UI 经常变但目标不变的场景

## 09 顾问模式

用 Sonnet 做执行，在需要战略决策时调用 Opus 4.7 做顾问。

```python
tools = [
    {
        "type": "advisor_20260301",
        "name": "advisor",
        "model": "claude-opus-4-7"
    },
    {
        "type": "computer_20251124",
        "name": "computer",
        "display_width_px": 1280,
        "display_height_px": 720
    }
]
```

整个咨询过程在服务端一次请求内完成，不需要额外的客户端往返。

**小问题：** 长对话中 Sonnet 容易忘记自己有顾问可用。Anthropic 建议每隔 20 轮左右提醒一下「记得你还有 advisor 工具可以用」。
