# AI 知识库技术演进拆解：从 RAG 到 NotebookLM，再到 LLM Wiki

> 来源：叶小钗｜AI训练营10期，6月底开班  
> 发布时间：2026年5月14日  
> 原文链接：https://mp.weixin.qq.com/s/UtN4_bhOYBV3OnYIUGN4LQ

## 核心观点

### NotebookLM 的本质

NoteBookLM 是一款基于用户上传资料的 AI 笔记与研究助手，核心逻辑是：**只根据你喂给它的资料来回答**，极大降低了 AI 幻觉。

严格来说，一套好的 AI 知识库大概率是一套 RAG 系统，但在 NoteBookLM 这套工具上又看不到 RAG 存在的痕迹。

### Karpathy 的 LLM Wiki 理念

传统 RAG / NotebookLM / ChatGPT 文件上传，大多是：
> 上传资料 → 查询时召回相关片段 → 临时综合回答

但问题是：每次问问题，模型都在重新从碎片里**现拼答案**，知识没有持续沉淀。

Karpathy 提出的 LLM Wiki 则是：
> 原始资料不动 → LLM 读取资料 → 增量维护一个结构化 Wiki → 不断更新实体页、主题页、交叉引用、矛盾点和综合结论

**不是查询时拼答案，而是提前把知识编译成一个可持续演化的知识结构。**

## NotebookLM 的技术路径

### 整体技术架构（七层）

```
用户上传资料 Sources
↓
文档解析与结构化
↓
多粒度索引
↓
Retrieval and Ranking
↓
长上下文装配
↓
Gemini 基于资料生成
↓
引用、摘要、问答、报告、音频
```

### 关键：NotebookLM 本质也是 RAG

但不是低配 RAG，而是一套被长上下文、检索排序、上下文工程、结构化笔记、source grounding 包装过的高阶知识库系统。

传统 RAG 链路暴露很明显：`分块 → 向量化 → 检索 → 重排 → 拼上下文 → 回答`

但在 NotebookLM 的产品体验里，看不到知识库配置、TopK、向量召回、重排模型、score 阈值这些东西——因为 Google 把这套工程链路**自动化、黑盒化**了。

## RAG 产品化的核心思路

**传统 RAG 的问题**：整套链路非常工程化，也非常折磨人。

单说"怎么分块"这个问题：
- 按固定长度分块，可能会把完整观点切碎
- 按标题分块，遇到结构不清晰的文档又会崩
- 按段落分块，可能 chunk 太碎，语义不完整
- chunk 太大，召回不准
- chunk 太小，回答没有上下文

**产品化的解法**：把各种复杂的工程配置项变成系统默认能力。

从用户视角，只需要关心三点：
1. 我上传了哪些资料？
2. 我想问什么问题？
3. 我是否能回到原文核查？

至于背后的：怎么解析？怎么清洗？怎么切块？怎么建索引？怎么召回？怎么排序？怎么拼上下文？怎么做引用？**全部由系统完成。**

## 七层技术架构详解

### 一、Source 接入

Notebook 的对象结构：
```
Notebook
├── Sources：用户上传的资料
├── Notes：用户笔记
├── Conversations：历史对话
├── Indexes：索引系统
└── Settings：Notebook 配置
```

核心产品抽象：**把资料当成可以被引用和追溯的事实来源。**

### 二、文档理解（最关键、最难的一层）

传统 RAG 第一步是 PDF 转文本，但真正有价值的不仅是正文，还有：
- 标题层级、章节关系、页码
- 表格、图片、图注
- 目录、脚注、参考文献
- 时间、作者、来源

**把原始资料还原成机器可处理、可切分、可索引、可引用的结构化文档。**

如果这里做得很差，后面就只能在一堆碎片里**硬搜**。

> "就算我们知道了这是 NoteBookLM 技术路径的核心，但我们其实也是做不出来的。因为 Google 本来在信息搜索这块就是世界 T0 级别的..."

#### 文档理解的核心：还原

不是把文档转成文本，而是尽可能还原文档原本的知识结构。

**表格处理**也是关键——更好的做法是把表格转成结构化对象：
```json
Table
├── title: 经营指标
├── columns: 年份 / 收入 / 毛利率
├── rows: 2023 / 100亿 / 35.2%
├── rows: 2024 / 120亿 / 38.4%
└── page: 46
```

#### 混合策略

```
结构化长文档 → 文档树 / PageIndex / 树搜索
碎片化知识 → 向量检索 + BM25 + Rerank
表格型资料 → 表格结构化 + 字段查询
多模态资料 → OCR / ASR / 图表理解 / 多模态摘要
```

### 三、Chunk（多粒度）

传统 RAG 喜欢讨论 chunk size，RAG 产品不会只有一种 chunk，而是维护多种粒度：

- Source 级别
- Chapter 级别
- Section 级别
- Paragraph 级别
- Chunk 级别
- Sentence 级别

**关键原则：chunk 不能和原文结构脱钩。**

---

## 核心结论

> **文档理解可能是整个 RAG 产品里最难的一层。**
> 
> 很多人以为 RAG 的难点在向量库、在 embedding、在 rerank，但这些东西更多是**检索算法层**的问题；真正决定知识库上限的，往往是前面的资料有没有被正确理解。
>
> 因为他们本质上都是在**已经被处理好的材料**上做优化。
