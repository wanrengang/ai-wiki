## 为什么可观测性成了AI智能体的生死线

### 痛点场景

很多 OpenClaw 用户都经历过的场景：

1. **API账单突然翻了三倍**，但不知道钱花在哪了——是某个 Agent 陷入了思维循环？上下文滚雪球导致 Token 膨胀？还是模型选错了？

2. **"诡异行为无法复现"** — Agent 做了一个奇怪的决策，用户来投诉，翻日志只看到一行 `Error: Context Deadline Exceeded`。30秒里 Agent 在干什么——完全黑盒。

### 根源问题

OpenClaw 处理一个简单请求，内部要经历：
- 消息接入
- 会话路由
- 上下文组装
- LLM 推理
- 工具调用
- 结果返回

任何一步出问题都会导致输出偏离。**没有可观测性，AI Agent 就是在盲飞。**

---

## OpenTelemetry怎么接进OpenClaw的

### OpenTelemetry（OTel）核心能力

三类数据：
- **Traces（链路追踪）**
- **Metrics（指标监控）**
- **Logs（日志记录）**

### 接入架构（三层）

**底层：diagnostics-otel 插件**
- OpenClaw 官方内置的 Metrics 采集插件
- 负责系统运行态指标的实时采集——消息队列深度、会话状态、网关健康度、内存压力
- 不需要额外安装

**中间层：OTLP 协议通道**
- 所有遥测数据走 OTLP（OpenTelemetry Protocol）标准格式上报
- 关键设计：数据可以发给任何兼容 OTel 的后端，**不锁定在某个特定平台**

**上层：可观测性后端**
三类选择：
1. ClawMetry 本地仪表盘
2. 腾讯云/火山引擎等云平台方案
3. 自建 Jaeger + Grafana 方案

### 安全设计

**OpenClaw 默认不暴露原始 prompt**。开发者能看到调用链和成本结构，但 prompt 内容不会被上报。

### GenAI 语义规范标签

遵循 OTel 的 GenAI 语义规范：
- `gen_ai.system` — 标注模型提供商
- `gen_ai.request.model` — 标注具体模型
- `gen_ai.usage.input_tokens` — 记录输入 Token 数
- `gen_ai.response.finish_reason` — 记录结束原因

## 三种接入方案

### 方案一：ClawMetry 本地仪表盘（三分钟上手）

**安装**：`pip install "clawmetry[otel]" && clawmetry`

**功能**：
- 实时事件流（Agent 的每一步操作以动画形式展示）
- Token 和成本追踪（按天/周/月汇总）
- 会话列表（哪个渠道、哪个模型、花了多少 Token）
- LLM 上下文检查器（系统提示词占用多少 Token、历史对话占多少、工具定义占多少）
- Agent 运行时时间线（每轮对话中 LLM 调用和工具执行的耗时分布）

**优点**：零配置，装完就能用

**局限**：
- 本地开发工具，不支持多实例集群监控
- 没有告警能力
- 数据不持久化（关掉就没了）

---

### 方案二：火山引擎 TLS（一行命令接入云平台）

字节跳动火山引擎日志服务 TLS 在 4 月 23 号推出专门针对 OpenClaw 的全景观测方案，主打**零侵入、一键接入**。

**前提**：OpenClaw 版本不低于 2026.3.8，开通 TLS 服务，准备好 AK/SK

**四个观测大盘**：

1. **成本分析大盘** — 总调用次数、总 Token 消耗、总费用，按模型、Provider、Agent、主机等多维度拆解

2. **运维分析大盘** — Gateway 异常按类型分类（配置异常、WebSocket 异常、工具调用异常），多实例对比

3. **性能分析大盘** — 模型调度端到端延迟、消息队列处理延迟、Webhook 接收速率、会话卡死检测

4. **安全审计大盘** — 高危行为追溯（exec 执行危险命令、写入敏感路径）、鉴权失败统计、配置变更留痕

**优势**：门槛低、覆盖全
**局限**：绑定在字节生态内

---

### 方案三：腾讯云双插件协同（企业级全链路追踪）

目前公开资料中最深入的方案，"双插件协同"架构。

**第一个插件**：`openclaw-tencent-plugin`
- 专做 Trace 链路追踪
- 深度适配 OpenClaw 的 Hook 机制
- 为每次请求生成带父子关系的完整调用链：
  - `enter_openclaw_system`（请求入口）
  - → `invoke_agent`（Agent 执行）
  - → `chat`（LLM 推理）
  - → `execute_tool`（工具调用）
- 每个环节都生成专属 Span 节点

**第二个插件**：`diagnostics-otel`（内置）
- 负责 Metrics 指标采集

**腾讯云方案优势**：
- 链路追踪粒度最细——每次工具调用的参数、返回结果、执行耗时、是否报错都有记录
- Token 消耗拆解最精细——区分输入 Token、输出 Token、缓存读写 Token

**局限**：双插件架构维护复杂度高，适合已有腾讯云基础设施、对可观测性要求高的企业团队

## 行业影响

### 为什么现在推可观测性？

OpenClaw 的目标用户群体正在发生变化：
- 半年前主要是尝鲜的个人开发者
- 现在越来越多线上跑业务的企业团队在用

GitHub 上 36 万颗星、日活 Issues 和 PRs 各 500 条的社区活跃度，已经不能只靠极客热情支撑了。

### 赛道判断

当 AI Agent 从"能跑就行"走向"稳定可靠地跑"，可观测性就从 **nice-to-have** 变成了 **must-have**。

腾讯云和火山引擎几乎同时推出 OpenClaw 专属可观测方案，说明云厂商在押注 AI Agent 基础设施，而可观测性是第一块拼图。

**黑箱时代该结束了。**

开发者要的不只是一个聪明的 Agent，更要一个：
- **看得见**的 Agent
- **控得住**的 Agent
- **算得清成本**的 Agent

这才是 AI 智能体走向生产环境真正需要的东西。
