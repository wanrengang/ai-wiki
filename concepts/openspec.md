# OpenSpec

> 用 Spec 管需求，让 AI 编码不再失控

## 核心思路

不是写给人类看的文档，而是写给 AI 看的**契约** — 格式固定、结构清晰、双方先达成共识再动手。

解决 Vibe Coding（跟着感觉写代码）带来的需求模糊、接口对不上、返工率高的问题。

## 架构

项目根目录下就两个文件夹：

```
specs/     # 记录系统当前状态：模块、接口、数据模型、业务规则
changes/   # 每次变更的增量，含三份文件
```

### changes/ 三件套

| 文件 | 作用 | 解决什么问题 |
|------|------|-------------|
| `proposal.md` | 为什么要做（背景、目标、成功标准） | 方向对不对 |
| `design.md` | 怎么做（技术方案、接口设计、数据流） | 方案行不行 |
| `tasks.md` | 具体做什么（可执行任务 + 验收标准） | 落地执行 |

## 核心优势

### 1. 增量规范
- 用 `ADDED / MODIFIED / REMOVED` 标记变更，不重写整个文档
- 像 Git 管代码一样管规范：变更 → 合并 → 存档
- 变更完成后移到 `archive/`

### 2. 工具无关
- 本质就是一组 Markdown 文件，不绑定任何平台
- 支持 30+ 工具：Claude Code、Cursor、Codex CLI、通义灵码……
- 今天用 Claude Code 开发，明天换 Cursor，规范无需迁移

### 3. 四命令工作流
```
/opsx:propose  创建提案
/opsx:apply    应用变更
/opsx:sync     同步状态
/opsx:archive  归档
```

## 实测效果

- Token 消耗降 **30%-50%**
- 返工率降 **60%+**

## 局限

- 小改动走完整流程偏重，需判断何时轻量处理
- 规范需跟着代码同步更新，否则会脱节
- 写好 proposal/design 本身需要经验，有学习曲线

## 上手

```bash
npx openspec init
```
无需 API Key，无需外部依赖。
