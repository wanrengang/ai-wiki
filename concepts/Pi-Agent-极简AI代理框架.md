# Pi Agent：一个和Claude code 不一样的AI工具，为什么会有这么多人用？

> 来源：微信公众号「创见AI实验室」  
> 日期：2026年5月28日 20:03

## 核心观点

以代码为中心设计的 AI 工具，在非编程场景下反而会变成枷锁。它的系统提示词、内置工具、安全边界，全都在无形中把对话拉回"工程思维"。而人们只是想要一个干净的、不被预设功能绑架的交互环境。

## Pi Agent 是什么

Pi 是一个 GitHub 上 56,600 多 Star 的极简代理框架。它并不是用来"取代" Claude Code 的，而是用来补全另一个维度的需求：当你的目标不是交付代码，而是让 AI 纯粹地理解并执行你的意图时，Pi 可能是目前最清爽的选择。

### 体验感触

1. 这是一个开放的AI Agent，不止编码，它能做什么取决于你想让它干什么
2. 这是一个开箱极简、成长性和定制化极高的 AI Agent。你可以把它改成任何你想的样子
3. 需要什么装什么，而不是从一堆 skill、mcp 等工具里面选用什么，这样在使用的时候会更加清晰地知道 Pi 能干什么

## 编程工具的局限

无论是 Claude Code、Open Code 还是 Pi，本质上都是 **Harness（工作台）**。模型提供智能，工作台负责管理提示词、上下文、工具和记忆。

问题在于，为了服务好"编程"这一个目标，主流工作台悄悄注入了大量预设：
- Claude Code 的系统提示词有上万 Token，内置了十几种工具、子代理逻辑、权限校验
- Codex 的上下文里塞满了代码补全、仓库分析、安全规则

**这些设计让它们在写代码时无比强大，但一旦离开这个预设场景，就变成了"噪音"。**

另一个问题是模型绑定。Claude Code 只认自家模型，为了换模型而更换整个工作台，就像为了换轮胎而买辆新车。

## Pi 的解法：把工作台"格式化"，只留一个空壳

Pi 走了一条截然不同的路：**它几乎不预设任何关于"你要做什么"的假设。**

- 系统提示词短到可以忽略
- 没有内置的子代理、计划模式
- 连常见的确认弹窗都省了

### 灵活的模型切换

Pi 原生支持 15 个以上的供应商，从 Anthropic、OpenAI 到国内的 Kimi、MiniMax、小米 MiMo，一个 `/model` 命令就能实时切换。

### 角色扮演能力

想做什么角色直接写一段系统提示词，Pi 会完全遵照这个设定来回答，不会突然跳出"需要我把分析结果保存为 CSV 吗"这样的代码思维。

### 开放的扩展生态

3100 多个第三方扩展包已经把 Plan Mode、MCP、权限门控等能力变成了可选项，需要时再装，不用时它们绝不占一丝上下文。

## 对比：厨师 Agent vs 代码审查员

同一个工具，通过不同的上下文注入，既能当厨师，也能当程序员。

**Claude Code**：输入指令后，它很高效，但会让用户指定项目目录、建议用 GitHub Actions 定期更新——很专业，但用户要的不是软件项目，而是一段有烟火气的对话。

**Pi**：直接写一段角色描述作为系统提示词，它就会完全扮演好这个角色，不提任何文件、任何命令。

## 总结

Pi 填补了"非典型"需求的空白：
- 当你需要跨模型实验
- 当你想让 AI 扮演特定角色
- 当你渴望一个不被工具预设绑架的对话环境
- 甚至当你只是单纯地跟一个干净的 AI 聊聊天

**"有很多 agent harness，但这个是属于你的。"**
