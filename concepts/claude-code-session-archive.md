# Claude Code 会话归档系统

> 基于 Obsidian 的本地归档方案，jsonl 自动转可读 markdown，三层架构按需取，零向量库零后台进程。

## 核心问题

Claude Code 桌面端四个结构性缺陷：

| 问题 | 说明 |
|------|------|
| **无会话入口** | 重启后侧边栏不显示历史会话，数据在 `~/.claude/projects/` 但无法访问 |
| **无 resume 命令** | `/` 开头的斜杠命令在桌面端不可用 |
| **md 档案不完整** | AI 总结的 md 是二次加工，关键细节、被否决方向全部丢失 |
| **jsonl 不可读** | 机器格式，一行一个 JSON，人眼无法直接阅读 |

**根本原因**：数据还在，但既没入口又不可读。向量索引解决不了这些。

## 三层架构

```
索引层  MEMORY.md        ~200 行，永远自动加载
策划层  memory/*.md      精炼的反馈/偏好/项目结论
原始层  sessions-md/     完整对话归档，按需取
```

| 层级 | 作用 | 类比 |
|------|------|------|
| **原始层** | 完整归档，按需取 | 整个库藏 |
| **策划层** | 精炼的反馈/偏好/结论 | 分类摆放的书架 |
| **索引层** | ~200 行，自动加载 | 借书卡 |

> 三层不是冗余，是递进——原始层管「全」，策划层管「精」，索引层管「轻」。

## 核心实现

六个 Node 脚本，约 800 行代码：

1. **jsonl 转 md** — `archive-convert.js`：读取 jsonl，按对话格式写成 md，工具调用默认过滤
2. **双向同步** — `session_id` 机制，已有的 md 只生成不覆盖，手动编辑不会被下次同步抹掉
3. **预分类** — 聊完说「归档」，当前 AI 直接写分类，今天几百 token 换明天省几千 token

### session_id 机制

```yaml
---
session: f75d9cf5-cc4e-4ca8-a5ad-03a6d19b5454
title: "会话归档系统建设"
date: 2026-04-17
folder: claude-system
---
```

session_id 作为唯一绑定，在 Obsidian 里移动/重命名/删除都不影响。

## 桌面端触发方式

用 AI 的 Bash 工具作为中间层，自然语言触发：

| 自然语言 | 实际效果 |
|----------|----------|
| 「归档」/「再见」/「下次继续」 | AI 预写当前会话分类（最常用） |
| 「上次聊 X 的会话」 | AI 读摘要文件，列出候选会话 |
| 「合并 A B C 会话」 | AI 按时间线合并，原文软删除 |
| 「删掉 XX 对话」 | md + jsonl 一起移到 trash |
| 「重建归档」 | 强制重生成所有 md（应急用） |

**日常只需要记住三条**：
1. 聊完说「归档」
2. 要彻底删了说「删掉 XX」
3. 想重建全部说「重建归档」

## 四层检索机制

| 层级 | 消耗 | 说明 |
|------|------|------|
| **L0** | 0 token | 直接打开 Obsidian 自己翻 |
| **L1** | ~5K | AI 读项目摘要文件，列出候选会话 |
| **L2** | ~2K | AI 读会话头部元数据，确认目标 |
| **L3** | ~1-2K | AI 用 Grep 在会话原文里搜关键词 |

典型场景（继续聊上次某个项目的接口问题）：
1. AI 读项目摘要（~2K）→ 列 3 个相关会话
2. 用户选一个，读会话头部摘要（~1K）确认
3. 如需具体细节，再 Grep 定位（~2K）

**总计 5K token，覆盖「挑 → 确认 → 看细节」全流程**

## 日常成本

AI 自动跑两步（透明）：
- 扫昨天有没有漏归档的会话（有就自动转 md）
- 反向同步 Obsidian 变更

| 情况 | token 消耗 |
|------|-----------|
| 无变化 | ~10 |
| 有删改 | ~30 |
| 有新会话未归档 | ~100 |

## 五条不妥协原则

1. **原文永远不碰** — 所有脚本只写 md，不改 jsonl
2. **索引可以重建** — manifest 丢了不可怕，扫一遍 md 就能重建
3. **删除都是软的** — 移到 trash 目录，永不自动清理
4. **分类由当前 AI 写** — 同一件事，现在做比以后做又准又省
5. **系统不跑时不花一分 token** — 每一步都清楚、可见、可回滚

## 设计原则

> **痛点要说到最具体。** 最开始以为问题是「Claude 没记忆」，真正说清楚才发现，是「数据既没入口又不可读」。前者是抽象需求，后者是具体数据问题。抽象需求会被复杂工具吸引，具体数据问题往往有最简解法。

> **好的架构是分层的。** 原始、策划、索引三层分工，每一层只回答一个问题：要全、要精、要轻。混在一起的系统一定会在某个维度失衡。

> **最好的工具不求智能，求可预测。** 不做向量检索，不调 AI 摘要，不跑后台进程。每一个动作都清楚、可见、可回滚。可预测性就是价值本身。

> **工具的使用成本，不该由使用者承担。** 最好的 AI 协作系统，最不该依赖 AI；最好的桌面端工具，最不该依赖 CLI。
