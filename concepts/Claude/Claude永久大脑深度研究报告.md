## 一、核心概述

Anthropic 正在为 Claude 构建一套完整的「持久记忆 + 永不下线 Agent」体系，核心包含三个产品：

| 产品 | 定位 | 状态 |

| **Memory Files** | 下一代记忆架构（文件系统式） | 即将推出 |
| **Dreams / Auto Dream** | 记忆整合机制（REM睡眠仿生） | 灰度测试中 |

## 二、Memory Files：内置个人 Wiki

### 2.1 现有「经典记忆」的局限性

Claude 现有的"经典记忆"本质是一张**滚动便签纸**：
- 将用户信息压缩进单一摘要条目
- 信息量一大就开始"溢出"
- 旧的被新的覆盖，重要的被琐碎的淹没
- 记住了昨天点的咖啡口味，却忘了上个月花三小时讨论的产品架构

### 2.2 Memory Files 的突破

Anthropic 给 Claude 造了一个**内置个人 Wiki**，彻底重构记忆系统：

**核心优势：**

| 能力 | 说明 |
|------|------|
| **容量无限** | 基于文件系统，理论可无限扩展，不再受单一摘要长度限制 |
| **精准检索** | 聊代码时只调取技术文档，不是"全量记忆灌入"，而是**按需检索** |
| **用户可控** | 可像编辑 Wiki 一样随时浏览、修改、删除任何记忆文件 |

**技术本质**：Anthropic 把 OpenClaw/Hermes 等长时运行 AI 智能体已有的架构，做成了**消费级产品的原生功能**。

### 2.3 双模记忆系统

据 TestingCatalog 报道，Anthropic 正在测试一套**双模记忆系统**，用户可在两种模式间自由切换：

1. **经典模式（Classic）**：现有的单一摘要模式
2. **文件模式（Memory Files）**：文件系统式多文档模式

## 三、Dreams / Auto Dream：REM 睡眠仿生机制

### 3.1 问题：Auto Memory 的衰减困境

Auto Memory 是一个突破——Claude 终于会自己做笔记了。但 **20+ 轮对话后**，这些笔记会变成一团乱麻：

- 矛盾的条目堆积
- 相对日期（如"昨天"）失去意义
- 过时的调试方案引用已被删除的文件
- 笔记本从"帮助记忆"变成了"噪声来源"

### 3.2 Dreams 的灵感来源

**命名是故意的，比喻惊人地准确。**

人类神经科学中，REM 睡眠阶段大脑并没有闲着——它在：
- 回放白天的经历
- 强化重要的记忆连接
- 丢弃无用的噪声信息
- 将短期记忆整合为长期记忆

**不经历 REM 睡眠的人，记忆力会显著下降。**

Anthropic 把同样的逻辑搬到了 Claude 身上。

### 3.3 Auto Dream 的四阶段工作流程

当 Claude 闲下来时，Dreams 自动启动，对记忆文件深度整合：

#### Phase 1: Orientation（定向）
Claude 读取当前记忆目录，扫描 MEMORY.md（索引文件），建立当前记忆状态的 mental map。

#### Phase 2: Gather Signal（收集信号）
在会话 JSONL 文件中进行**窄搜索**，而非通读每个会话（成本太高）：
- 用户纠错时刻
- 显式保存（"记住这个"）
- 反复出现的主题
- 重要决策

#### Phase 3: Consolidation（整合）
核心阶段，执行关键维护：
- **日期规范化**：「昨天决定用Redis」→「2026-05-15 决定用Redis」
- **删除矛盾事实**：如果三周前已从 Express 切换到 Fastify，删除旧的"API使用Express"条目
- **删除过时记忆**：关于已被重构删除的文件的调试笔记
- **合并重复条目**：三次会话记录了相同的构建命令quirk，整合为一条干净条目

#### Phase 4: Prune and Index（剪枝与索引）
更新 MEMORY.md 索引文件，保持在 **200 行以内**（这是启动时加载的阈值）：
- 删除已不存在的主题文件指针
- 添加新创建的主题文件链接
- 解决索引与实际文件内容的矛盾
- 按相关性和时间重新排序

### 3.4 触发条件

两个条件**同时满足**才触发整合周期：
- 距上次整合已过 **24 小时**
- 自上次整合以来发生了 **超过 5 次会话**

双重门控防止了：
- 轻使用项目不必要的整合运行
- 高频短期会话的过度整合

**实测案例**：913 轮会话的记忆整合耗时约 **8-9 分钟**。

### 3.5 安全约束

| 约束 | 说明 |
|------|------|
| **只读模式** | 梦循环期间只能写记忆文件，不能修改项目代码、配置、测试 |
| **锁文件** | 防止两个 Claude Code 实例同时运行 Auto Dream，避免合并冲突 |
| **后台执行** | 可在 Claude Code 工作中运行，不阻塞会话 |

### 3.6 Claude Code 的四层记忆系统

Auto Dream 上线后，Claude Code 拥有了**四层记忆机制**：

| 层级 | 写入者 | 用途 | 时机 | 启动时加载 |
|------|--------|------|------|-----------|
| **CLAUDE.md** | 人类 | 指令和规则 | 手动编辑 | 完整文件 |
| **Auto Memory** | Claude | 项目模式和习得 | 每轮会话 | 无 |
| **Session Memory** | Claude | 会话摘要 | 自动 | 无 |
| **Auto Dream** | Claude | 记忆整合 | 每24h+5次会话 | 无 |

| 层级 | 存储位置 | 人类类比 |
|------|----------|----------|
| CLAUDE.md | ./CLAUDE.md | 指令手册 |
| Auto Memory | ~/.claude/projects/<project>/memory/ | 白天笔记 |
| Session Memory | ~/.claude/projects/<project>/<session>/session-memory/ | 短期对话回忆 |
| Auto Dream | 同 Auto Memory | REM睡眠整合 |

## 四、Conway：永不下线 Agent 平台

### 4.1 概述

Conway 是 Anthropic 正在测试的**杀手级永不下线 Agent 平台**。它将 Claude 从被动聊天机器人转变为 **7×24 小时自主运行的 AI 工作者**。

据 TestingCatalog 独家报道，Conway 以独立侧边栏选项出现，代码中描述为「Conway instance」——指向一个**独立环境**而非标准聊天界面。

### 4.2 核心技术架构

Conway 很可能不是全新架构，而是 Anthropic 将已发货的子系统打包成更明确的**持久 Agent 环境**：

| 子系统 | 功能 | 状态 |
|--------|------|------|
| **Cowork** | 桌面 Agent 运行时，基于 Claude Code | 已发布 |
| **Claude Code** | 开发工作专用执行面 | 已发布 |
| **Dispatch** | 手机到桌面任务传递，统一持久线程 | 已发布 |
| **Scheduled Tasks** | 定时执行 | 已发布 |
| **Computer Use** | 屏幕控制执行 | 已发布 |
| **Auto Mode** | 更安全的长时运行权限模型 | 已发布 |

### 4.3 关键功能

**1. 独立 Instance**
- Conway instance 意味着持久 Agent 容器概念
- 生命周期问题：何时唤醒、存活多久、什么持久、什么重置

**2. 扩展系统（.cnw.zip）**
- 支持自定义工具、UI 标签、上下文处理器
- 指向新的扩展包格式和第三方插件框架

**3. Webhook 触发**
- 这是最强的「永不下线」信号
- Agent 不再只等待用户提示，可被外部系统事件唤醒
- 从「我问你答」到「世界变化时行动」的跨越

**4. Chrome 直接连接**
- Claude in Chrome 可直接连接 Conway
- 浏览器作为持久动作面

**5. 通知系统**
- 任务完成、需批准、触发失败、环境离线、调度跳过
- 持久 Agent 需要紧凑的信令通道

**6. Epitaxy（尚未公开）**
- 可能是这个环境的操作界面

### 4.4 Auto Mode：永不下线的权限基础设施

Anthropic 于 2026 年 3月 24 日为 Claude Code 推出 Auto Mode。

这比看起来更重要——永不下线 Agent 如果每几分钟就停下来请求权限是没用的，但完全禁用权限又不安全。

**Auto Mode 的中间层方案**：
- Claude 可代表用户做出某些权限决策
- 分类器在工具调用运行前筛选
- 安全操作自动进行
- 风险操作被阻止或升级

### 4.5 尚需解决的五项难题

| 难题 | 说明 |
|------|------|
| **1. 触发信任** | Webhook 触发 Agent 强大但也制造攻击面：伪造事件、重放事件、毒化载荷、通过外部服务升级 |
| **2. 扩展隔离** | 第三方逻辑不能悄悄危害 instance。需要强答案：包信任、安装权限、工具作用域、网络边界、更新行为 |
| **3. 状态卫生** | 持久 Agent 累积上下文，累积的上下文会变坏。需要强机制：记忆剪枝、项目边界、陈旧状态检测、触发特定上下文窗口 |
| **4. 人类控制** | Agent 越持久，人类控制越重要。需要：暂停/恢复、紧急终止、审计追踪 |
| **5. 对齐与安全** | 永不下线 Agent 如果行为偏离目标，影响远超短暂会话。需要持续对齐监控 |

### 4.6 Conway vs OpenClaw 安全对比

文章原文指出：
> "OpenClaw 直接对标，但 Conway 是 Anthropic 原生的，直接跑在 Claude 上面，安全性和整合度完全不在一个量级。"

OpenClaw 头两个月就被发现 **9 个 CVE**，**4.2 万**暴露实例在公网裸奔。

**Conway 的安全优势**：
- 跑在 Anthropic 托管云基础设施上
- 扩展必须显式安装
- Webhook 可逐服务开关
- 浏览器集成走 Claude 权限模型

## 五、竞争格局：2026 三大巨头记忆系统对比

| 维度 | ChatGPT Memory | Claude Memory | Gemini Memory |
|------|----------------|---------------|--------------|
| **功能名称** | Memory | Memory | Memory（原 Saved Info） |
| **自动提取** | 自动 | 自动（每24h） | 自动 |
| **手动添加** | 支持（"记住xxx"） | 不支持 | 聊天中可纠正 |
| **容量** | Saved Memories 有上限，Chat History 无上限 | 有限 | 有限 |
| **用户可编辑** | 查看、删除、搜索、排序 | 查看、编辑、删除 | 查看、删除 |
| **跨平台导入** | 不支持 | 支持（从 ChatGPT） | 支持 |
| **记忆透明度** | 中等 | 相对高 | 低 |
| **记忆结构** | 非结构化文本条目 | 结构化叙事分类 | 简短条目 |
| **过往对话引用** | 引用 Chat History | Chat Search（RAG） | Past Chats |
| **记忆展示** | 纯文本列表 | 分段叙事（4类别） | 简短条目 |
| **付费墙** | Plus/Pro | Pro/Max/Team/Enterprise | Google AI plan 用户优先 |
| **隐私说明** | 可选用于模型训练 | 不用于训练 | 导入数据用于训练 |

### 各家路线

| 玩家 | 路线 | 记忆来源 |
|------|------|----------|
| **ChatGPT** | 个人助理路线 | Memory Sources：保存记忆/聊天历史/自定义指令/文件库/Gmail |
| **Gemini** | 谷歌生态路线 | 与 Gmail/Drive/Calendar 深度打通，来自真实生活数据 |
| **Claude** | 文件系统+自主进化路线 | Memory Files + Dreams + Conway 闭环 |

### Claude 的差异化优势

1. **结构化叙事记忆**：Claude 的记忆不是简单条目列表，而是分层、时间线感知的用户画像（工作背景→个人偏好→当前重点→历史上下文）
2. **跨平台导入**：支持从 ChatGPT 导入记忆，这是 Claude 独有的优势
3. **RAG 对话搜索**：Chat Search 提供清晰的搜索机制、引用来源和可点击链接
4. **Auto Dream 维护**：唯一具备自动记忆维护机制（REM睡眠仿生）

## 六、关键技术洞察

### 6.1 为什么 Memory Files 此时推出？

**为 Conway 铺路。**

永不下线的 AI 智能体最需要的是：不是一条随时被覆盖的滚动摘要，而是一套可以无限扩展、按需检索、自动维护的持久记忆系统。

- **Memory Files** → 提供存储架构
- **Dreams** → 提供维护机制
- **Conway** → 提供永不下班的运行时

三者叠加 = 从记忆到反思，再到行动的完整闭环。

### 6.2 Claude 的记忆演进路径

```
阶段1: 无记忆（每次对话从零开始）
    ↓
阶段2: 经典记忆（单一滚动摘要）
    ↓
阶段3: Auto Memory（会话级自动笔记）
    ↓
阶段4: Memory Files + Dreams（文件系统 + 自动维护）
    ↓
阶段5: Conway（永不下线 Agent）
```

### 6.3 企业落地效果

据 buildfastwithai.com 报道：
- **Rakuten** 的 Claude Agent 首次处理错误率暴降 **97%**，成本降低 **27%**，延迟降低 **34%**
- 原因：这些 Agent 记住了它们犯过的每一个错误

## 七、风险与挑战

### 7.1 技术风险

| 风险 | 描述 |
|------|------|
| **状态衰减** | 持久 Agent 累积上下文会变坏，需要强记忆剪枝机制 |
| **扩展隔离** | 第三方扩展可能悄悄危害 Agent 安全 |
| **Trigger 安全** | Webhook 触发存在伪造、重放、毒化攻击面 |
| **上下文膨胀** | 长期运行会导致上下文窗口不断膨胀 |

### 7.2 竞争风险

| 风险 | 描述 |
|------|------|
| **ChatGPT 先发优势** | ChatGPT Memory 已积累大量用户习惯数据 |
| **Gemini 生态优势** | Gmail/Drive/Calendar 深度整合带来真实生活数据 |
| **OpenClaw 等开源竞争** | 本地运行、隐私优先，Unix 式模块化设计 |

### 7.3 安全与隐私风险

| 风险 | 描述 |
|------|------|
| **数据驻留** | 记忆系统如何满足不同地区数据驻留要求 |
| **记忆泄露** | 持久记忆被未授权方访问 |
| **对抗性记忆** | 用户或第三方故意植入错误记忆 |

## 九、结论

Claude 的「永久大脑」体系代表了 Anthropic 在 AI 记忆和 Agent 架构上的**系统性押注**：

1. **Memory Files** 将记忆从「滚动便签」升级为「文件系统」，突破容量和精准度瓶颈
2. **Dreams/Auto Dream** 引入仿生记忆维护机制，解决了 Auto Memory 的衰减困境
3. **Conway** 将多个已发货子系统整合为统一的永不下线 Agent 平台

**核心优势**：
- 消费级产品体验（原生集成 vs 第三方插件）
- 系统性安全设计（Anthropic 基础设施 vs 开源方案）
- 结构化记忆架构（Wiki 式 vs 简单列表）

**核心挑战**：
- 发布时间未定（Memory Files 在 Beta，Conway 尚未官宣）
- 企业落地效果待验证
- 竞争压力持续增大（ChatGPT Memory 先行、Gemini 生态优势）

Anthropic 的策略很清楚：**不是在追赶对手，而是在定义下一代 AI 记忆和 Agent 的标准。**
