# 从0开发大模型的17种Agent架构演进详细拆解

> 来源：腾讯技术工程
> 作者：xzhou（公众号《周末程序猿》）
> 日期：2026年5月18日

## 核心观点

Agent architecture 的本质不是 prompt engineering，也不是某个框架的 DSL，而是**控制流设计**。

17种Agent架构是从**简单到复杂、从单次生成到多Agent协作**的演进路径，关键不是模型回答是不是够好，而是：

- 状态有没有被正确建模
- 控制流有没有被显式表达
- 错误能不能被局部截断
- 副作用能不能被关进闸门
- 系统知不知道自己什么时候该停

## Agent 演化路径总览

1. 从**单次生成**到**反思闭环**
2. 从**反思闭环**到**工具交互**
3. 从**工具交互**到**观察-行动循环**
4. 从**局部决策**到**显式规划**
5. 从**无验证执行**到**验证驱动重规划**
6. 从**单agent**到**多agent编排**
7. 从**短期上下文**到**长期记忆系统**
8. 从**线性推理**到**搜索、模拟与涌现计算**
9. 从**能做事**到**可信任**

## 统一分析框架：六个固定问题

每种架构都用同一套问题拆解：

1. **它要解决什么问题？** 上一代架构哪里不够
2. **它的 State 是什么？** 新增了哪些字段，为什么必须存在
3. **它的拓扑是什么？** 线性链、循环、分叉汇聚、共享黑板、树搜索还是网格涌现
4. **它的 Router 怎么工作？** 固定边、条件边、动态调度、验证回路、人工审批
5. **它的失败模式是什么？** 架构最容易在哪个环节坏掉
6. **什么时候该升级到下一种？** 当前模式的能力边界在哪里

## 17种Agent架构一览

| 阶段 | 新增能力 | 一句话解释 | 代表架构 |
|------|---------|-----------|---------|
| 单次生成优化 | critique pass | 把"生成"拆成 generator + critic + refiner 三步 | Reflection |
| 与世界交互 | tool interface | 打破参数知识边界和上下文封闭性 | Tool Use |
| 基于观察持续行动 | observation loop | Thought → Action → Observation 滚动循环 | ReAct |
| 先生成控制流再执行 | explicit planning | 把控制流本身做成可审计对象 | Planning |
| 把验证接入主回路 | verification loop | 失败就回到重规划，而不是事后检查 | PEV |
| 把认知任务拆成角色 | role decomposition | 用流水线或图把角色串起来 | Multi-Agent |
| 把中间状态显式共享 | shared workspace | Controller根据黑板状态动态决定下一个该谁上场 | Blackboard |
| 把入口做成路由系统 | entry routing | 把任务路由到最合适的专家子agent | Meta-Controller |
| 用冗余换可靠性 | parallel redundancy | 多agent处理+聚合/投票 | Ensemble |
| 把历史状态纳入系统 | long-term memory | 向量库+图/KV，让系统记得住、查得到 | Episodic + Semantic |
| 把推理变成搜索 | search tree | 多条思路形成树，边展开边打分、剪枝 | ToT |
| 把行动前评估做成模拟 | counterfactual execution | 先在内部世界模型里预演 | Mental Loop |
| 把副作用关进闸门 | side-effect gating | dry-run + 审核通过后才允许执行 | Dry-Run |
| 把自我边界建模 | self-boundary reasoning | 知道擅长什么、不擅长什么，据此选择 | Metacognitive |
| 把质量改进做成循环 | iterative refinement loop | editor打分+writer改稿，高分样本沉淀 | Self-Improve |
| 去中心化计算 | emergence | 每个单元只跑简单局部规则，全局行为涌现 | Cellular Automata |

## 关键架构详解

### Reflection（最小质量闭环）

**核心：** 把生成过程拆成 generator + critic + refiner 三阶段

**State：** draft / critique / refined_code

**拓扑：** 纯线性

**关键洞察：** LLM 作为 critic 往往比作为 generator 更稳定

**失败模式：** 有 critique，但没有闭环，不能验证 refiner 是否真的修好了问题

### Tool Use（文本世界到结构化世界的跨越）

**核心：** 让系统突破上下文与知识截止日期的封闭性

**关键代码：** DuckDuckGoTools()、YFinanceTools() 等

**关键洞察：** 不是"会调用函数"，而是**文本控制流可以跨越到结构化世界，再返回文本世界**

**失败模式：** 失败通常来自"边界层"——工具名幻觉、参数类型错误、工具返回格式不对

### ReAct（Agent真正成形的地方）

**核心：** 只要 model 回复里还有 tool_calls，就回到 model 再跑一次——形成持续交互

**关键代码：** `reasoning=True`

**拓扑：** 最小闭环（回边）

**关键洞察：** `tool → model` 的循环边是整个 agent architecture 里最重要的回边之一

**失败模式：** 局部贪心——容易走弯路、重复搜索、陷入局部最优

### Planning（把控制流本身变成模型输出）

**核心：** 把"控制流"对象化——先被生成出来，然后才执行

**State：** Plan.steps、intermediate

**关键代码：** `Loop` step 表达"循环执行直到 plan 空"

**关键洞察：** 路由逻辑从"LLM 在 prompt 里决定下一步"变成了"一个数据结构是否还有剩余项"

**失败模式：** 过于乐观，plan 错了后面每一步都可能是错的

### PEV（Plan → Execute → Verify）

**核心：** 把执行视为"生成了一个待验证结果"，而不是"完成了一步"
