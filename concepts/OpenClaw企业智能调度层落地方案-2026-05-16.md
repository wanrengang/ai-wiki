# OpenClaw 企业智能调度层落地方案

## 背景与核心定位

### 痛点
- 企业内部系统孤岛：飞书/钉钉/SAP/用友/自研系统互相割裂
- 重复性工作耗时：查假/查工资/查流程都要登录不同系统
- 外部平台能力未打通：运营商（电信/移动/联通）能力未纳入企业工作流
- AI 智能体落地难：每个用户都部署一个 agent 不现实

### 核心定位
**OpenClaw = 自然语言驱动的"企业 Siri"**

```
┌─────────────────────────────────────────────────────┐
│                   企业用户群                          │
│          管理层 | 销售 | 研发 | HR | IT运维          │
└─────────────────┬───────────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────────┐
│         OpenClaw 企业智能调度层                      │
│    "统一入口 + 自然语言驱动 + 平台串联"              │
└────────┬──────────────────────┬────────────────────┘
         ▼                      ▼
┌─────────────────┐    ┌─────────────────────────────────┐
│    内部系统      │    │          外部平台/生态           │
│  • 飞书/钉钉    │    │  • 微信客服                     │
│  • SAP/用友 ERP │    │  • 中国电信能力（星小辰/智铃）  │
│  • GitHub/Jenkins│   │  • 腾讯云/阿里云                │
│  • 自研系统 API  │    │  • 运营商数据                   │
└─────────────────┘    └─────────────────────────────────┘
```

## 一、部署架构

### 1.1 单实例多用户模式

**核心误解澄清**：不需要每个用户部署一个 OpenClaw，一台服务器部署一份可以服务多人。

```
                    ┌──────────────────┐
                    │   OpenClaw       │
                    │   Gateway        │  ← 只需部署一份
                    │   (单台服务器)    │
                    └────────┬─────────┘
                             │
         ┌───────────────────┼───────────────────┐
         ▼                   ▼                   ▼
    ┌─────────┐         ┌─────────┐         ┌─────────┐
    │ 飞书    │         │ 钉钉    │         │ QQ/微信 │
    │ 用户A   │         │ 用户B   │         │ 用户C   │
    │ 用户D   │         │ 用户E   │         │ 用户F   │
    └─────────┘         └─────────┘         └─────────┘
```

**关键特性**：
- 每个用户有独立的 Session（聊天历史隔离）
- Skills 一次开发，所有用户共享
- Workspace 配置由管理员统一管理

---

### 1.2 智能体隔离级别

OpenClaw 支持多级隔离：

| 隔离级别 | 实现方式 | 适用场景 |
|---------|---------|---------|
| **Session 级** | `session.dmScope: per-channel-peer` | 共享智能体，普通办公 |
| **Agent 级** | 独立 hr-assistant 智能体 | **企业 HR 助手（推荐）** |
| **完全隔离** | 每个员工独立智能体 | 高管/敏感岗位 |

**推荐方案：Agent 级隔离**

```yaml
# ~/.openclaw/config.yaml
agents:
  list:
    - id: hr-assistant
      workspace: ~/.openclaw/workspace-hr
      channels:
        qqbot:
          enabled: true
        feishu:
          enabled: true
        dingtalk:
          enabled: true

session:
  dmScope: per-channel-peer  # 每个员工独立会话

bindings:
  # 所有员工都路由到 HR 智能体
  - agentId: hr-assistant
    match:
      channel: qqbot
      peer:
        kind: direct
  - agentId: hr-assistant
    match:
      channel: feishu
      peer:
        kind: direct
  - agentId: hr-assistant
    match:
      channel: dingtalk
      peer:
        kind: direct
```

## 二、路由机制

### 2.1 Binding 路由 vs 内容路由

| 方式 | 原理 | 支持情况 |
|------|------|---------|
| **Binding 路由** | 根据 sender ID/channel 等元数据匹配 | ✅ OpenClaw 标准方式 |
| **内容路由** | AI 分析消息内容决定路由 | ❌ 不支持 |
| **Delegation 委托** | Agent 收到后委托给其他 Agent | ✅ 适合 Agent 间协作 |

### 2.2 Binding 路由原理

```json
{
  "bindings": [
    // 员工A的消息 → HR智能体
    { "agentId": "hr-assistant", "match": { "channel": "qqbot", "peer": { "id": "A员工QQ号" } } },
    // 员工B的消息 → HR智能体
    { "agentId": "hr-assistant", "match": { "channel": "qqbot", "peer": { "id": "B员工QQ号" } } }
  ]
}
```

**路由流程**：
```
员工发消息
       │
       ▼
  OpenClaw Gateway 识别元数据
  (channel=qqbot, peer=员工QQ号)
       │
       │ 匹配 binding
       ▼
  hr-assistant 智能体
  (加载 HR 工作区)
       │
       ▼
  返回结果
```

```
                    ┌──────────────────────┐

                    └──────────┬───────────┘

   │ 飞书       │        │ 钉钉      │          │ 企业微信   │
   │ 员工A      │        │ 员工B     │          │ 员工C     │
```

### 3.2 员工无需安装任何软件

员工只需要在已有的通讯工具（飞书/钉钉/企业微信）里发消息，OpenClaw 就会响应。

| 功能 | 实现方式 |
|------|---------|
| 查年假/请假 | 调 HR 系统 API（用友/SAP）|
| 入职指引 | 查员工手册知识库 |
| 政策问答 | 查 FAQ 知识库 |
| 提交审批 | 调 OA 系统 API |
| 会议纪要 | 调日历 API + 录音转写 |

## 四、多用户隔离机制

### 4.1 隔离原理

```
                    ┌────────────────────────────┐
                    │   Session Pool (会话池)    │
                    │  ┌──────┐ ┌──────┐ ┌──────┐│
                    │  │员工A │ │员工B │ │员工C ││
                    │  │Session│ │Session│ │Session││
                    │  └──────┘ └──────┘ └──────┘│
                    │  每个用户独立上下文/记忆      │
                    └────────────────────────────┘
                                   │
                                   ▼
                    ┌────────────────────────────┐
                    │   Skills / Tools          │
                    │  • 飞书 API               │  ← 一次开发，所有用户用
                    │  • HR 系统 API             │
                    │  • 知识库检索              │
                    └────────────────────────────┘
```

### 4.2 TOOLS.md/USER.md 会被人乱改吗？

**不会！** 权限控制机制：

| 防护层 | 说明 |
|-------|------|
| 文件权限 | TOOLS.md 是服务器文件，普通用户无法直接编辑 |
| Tool Policy | 可配置禁止普通用户调用 exec/write 等危险工具 |
| Skill 管控 | 可限制普通用户能使用哪些 Skill |

```yaml
# 可选的 Tool Policy 配置
tools:
  policy:
    exec: deny        # 普通员工禁止执行命令
    write: deny       # 普通员工禁止写文件
    read: allow       # 允许读文件
```

## 六、优先落地场景

### 6.1 场景优先级

| 优先级 | 场景 | 成熟度 | 原因 |
|--------|------|--------|------|
| 🔴 高 | IT 运维智能助手 | 高 | API 完善，场景清晰 |
| 🔴 高 | HR 智能助手 | 高 | 需求强烈，复用度高 |
| 🟡 中 | 销售/客服支持 | 中 | 跨系统数据整合复杂 |
| 🟡 中 | 研发效能提升 | 中 | GitHub/Jenkins API 完善 |
| 🟢 低 | 管理层决策支持 | 低 | 数据源分散 |

### 6.2 IT 运维智能助手

**痛点**：监控告警多、人工响应慢、跨系统排查复杂

**实现**：
```
用户："服务器响应慢了，帮我看看原因"
  → 调 GitHub API 查最新部署记录
  → 调监控 API 看 CPU/内存/网络指标
  → 调日志系统查 Error 日志
  → 综合分析后给出结论 + 建议
```

## 八、关键挑战与应对

| 挑战 | 应对策略 |
|------|---------|
| **数据安全** | 本地部署 + 权限分级 + 操作审计 |
| **系统集成复杂度** | 先从 API 完善的系统入手（飞书/GitHub），逐步深入 |
| **用户 adoption** | 从高频痛点切入，让用户感受到"省时间" |
| **维护成本** | 培养"OpenClaw 管理员"角色，负责 SKILL 开发维护 |
| **AI 幻觉/错误** | 关键操作需人工确认后再执行 |

## 十、核心结论

1. **不需要每人部署一个**：一台服务器 + 一个 HR 智能体 + 会话隔离就能满足需求

2. **路由靠 Binding**：在配置里写好规则，OpenClaw 自动路由，不需要内容分析

3. **TOOLS.md 有保护**：普通员工无法修改配置文件，也无法执行危险操作

4. **最佳隔离级别**：Agent 级隔离，平衡了安全性和运维成本

5. **员工零门槛**：不需要安装任何软件，在已有通讯工具里发消息即可
