# MiroFish 群体智能预测引擎深度研究

> 研究日期：2026-05-31
> 来源：GitHub、官网、技术解析文章

## 一、项目概述

**MiroFish** 是由**中国科学技术大学大四学生郭航江**（网名"百夫"）开发的一款开源群体智能预测引擎，2026年3月登顶 GitHub Trending 全球榜首，两周内斩获 25,000+ Star。核心代码**10天完成**，采用"Vibe Coding"（大量依赖 Claude Code 等 AI 辅助编程）。

**融资情况：** 项目demo向陈天桥演示后24小时内，敲定**盛大集团3000万人民币**战略注资。

**前作：** 2025年下半年曾推出 BettaFish（AI舆情分析工具），一周内涨了2万+ Star，同样登顶 GitHub Trending。

**核心定位：** 不是"准确率95%的预测工具"，而是**决策沙盒**——给你一个低成本试错环境，让未来先在数字沙盘里演一遍。

## 二、技术架构详解

### 三层架构

```
┌─────────────────────────────────────────┐
│  交互层：输入种子材料 → 输出预测报告     │
├─────────────────────────────────────────┤
│  智能体层：数千个有独立人格的 AI Agent   │
├─────────────────────────────────────────┤
│  世界层：知识图谱 + 仿真引擎 (OASIS)     │
└─────────────────────────────────────────┘
```

### 第一层：世界层（从文本到知识图谱）

**核心技术：时序 GraphRAG**

依赖 Zep Cloud 的图谱引擎 Graphiti，流程：

```
种子文本 → LLM实体关系抽取 → 生成本体结构 → 构建知识图谱
```

**关键设计——动态失效机制：**
- 知识图谱不是静态的，随仿真推进旧关系会衰减、新关系会建立
- 比简单 RAG 检索更合理，现实世界的关系确实是随时间变化的

**实例：** 输入一篇关于美伊关系的学术报告，系统自动提取118个实体（伊朗核科学家、美国前国务卿、CNN媒体等），并建立 SUPPORTS、CRITICIZES、ALLIED_WITH 等关系。

### 第二层：智能体层（有人格的 Agent）

每个实体会被转化为一个**独立的 Agent**，每个 Agent 拥有：

| 组件 | 说明 |
|------|------|
| 全景经历 | 基于种子材料生成的背景故事 |
| 行为模式 | 决策倾向、沟通风格 |
| 社会关系 | 与其他 Agent 的关系网络 |
| 性格特征 | 影响决策偏好和表达方式 |

**关键：长期记忆系统**
- 通过 Zep Cloud 的时序记忆系统，Agent 能记住之前的交互历史
- 据此调整后续行为，实现"群体涌现"

**双平台并行仿真：**
- **短消息平台（类Twitter）**：信息传播快、碎片化、情绪驱动
- **长讨论平台（类Reddit）**：深度讨论、观点碰撞、社区极化

模拟信息在不同类型社交网络中的传播差异。

### 第三层：交互层（输入→输出）

```
种子材料（新闻/政策/小说）
  ↓
Graph Building：种子提取、记忆注入、GraphRAG构建
  ↓
Environment Setup：实体关系提取、人格生成、Agent配置
  ↓
Simulation：双平台并行仿真、动态时序记忆更新
  ↓
Report Generation：ReportAgent 分析后仿真状态，输出结构化报告
  ↓
Deep Interaction：可与任意Agent对话，深入探索
```

## 三、技术栈

| 组件 | 技术选择 | 说明 |
|------|----------|------|
| 后端 | FastAPI (Python) | 异步框架，适合 I/O 密集场景 |
| 前端 | Vue 3 | 可视化交互界面 |
| 仿真引擎 | OASIS (CAMEL-AI) | 学术验证的多智能体仿真框架 |
| 记忆系统 | Zep Cloud + Graphiti | 时序 GraphRAG |
| LLM | 阿里百炼 qwen-plus | 兼容 OpenAI SDK 格式 |
| 部署 | Docker Compose | 一键部署前后端 |

**运行环境：** Node.js 18+，Python 3.11-3.12
**服务端口：** 前端 3000，后端 API 5001

## 四、与 OASIS / CAMEL-AI 的关系

**OASIS（Open Agent Social Interaction Simulations）** 是 CAMEL-AI 团队开源的多智能体仿真框架，MiroFish 的仿真引擎基于 OASIS 做了轻封装。

MiroFish 的核心价值在**业务价值层**（预测引擎的输入输出体验、场景适配），底层仿真能力交给经过学术验证的 OASIS。

这个决策很务实：10天完成的项目，把精力集中在业务逻辑而非底层仿真引擎。

## 五、工程决策亮点

| 决策 | 选择 | 理由 |
|------|------|------|
| 仿真引擎 | 复用 OASIS | 学术验证，无需从头造轮子 |
| 记忆系统 | Zep Cloud SaaS | 开发效率高，免费额度够 MVP |
| 进度通知 | 前端轮询 | 仿真长时间运行，轮询比 WebSocket 更稳定 |
| 日志存储 | JSONL 文件 | 多 Agent 并发写入不需要锁，优先性能 |

## 七、"Vibe Coding"现象

MiroFish 背后值得关注的现象：**Vibe Coding**

> 概念由 Andrej Karpathy 提出：开发者不再逐行写代码，而是用自然语言描述需求，让 AI 生成代码，开发者负责审查和方向把控。

**启示：**
- AI 辅助编程的效率已到临界点：一个人+AI工具链，可以在极短时间内构建工程化程度不低的系统
- 核心竞争力在**系统设计**，不在写代码
- 开源 + AI 工具链 = **超级个体**：善于组合已有开源组件

**局限性：** Vibe Coding 目前适合快速验证和 MVP 阶段，在需要深度优化、高并发、安全审计的场景下仍需传统工程能力。

## 八、局限性分析

1. **预测准确性难以验证**
   - 结果高度依赖 Agent 人设质量和 LLM 推理能力
   - 同样的种子材料，不同 LLM 或 Agent 数量可能得到截然不同的结果
   - 定位是"决策沙盒"而非"准确预测"

2. **成本不低**
   - 每次仿真需要大量 LLM 调用（Agent数量 × 交互轮数 × 每轮推理）
   - 40轮以上的模拟成本不容忽视

3. **核心仿真引擎非自研**
   - OASIS 是 CAMEL-AI 团队的学术成果
   - MiroFish 核心价值在产品层面，不在底层技术
