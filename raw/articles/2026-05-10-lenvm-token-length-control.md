# LenVM：将长度建模提升到Token级别

**论文**: https://arxiv.org/abs/2604.27039
**代码**: https://github.com/eric-ai-lab/Length-Value-Model
**项目主页**: https://length-value-model.github.io/

## 核心问题

当前长度控制方法全都太粗糙：

| 现有方法 | 局限 |
|---------|------|
| 序列级惩罚 | 解码过程中对"还剩多少"毫无感知 |
| Prompt指令 | 本质是"祈求"模型遵守，无硬约束 |
| 预解码长度预测器 | 一次性判断，无法动态调整 |

共同局限：都在序列层面操作，而解码本身是逐token发生的。

## 核心方案：把剩余长度变成值函数

### 关键创新

将生成长度建模转化为强化学习中的价值估计问题：

> 给每个生成的 token 分配固定负奖励，折扣累加得到「剩余生成长度」的有界单调代理信号

### 数学构造

对每个非终止解码步 t，分配固定负奖励 r，折扣回报为：

```
R(t) = -γ^t / (1-γ)
```

其中 L 是序列总长度，γ ∈ (0,1) 是折扣因子。

### 三大关键性质

- **有界**：R(t) ∈ (-1, 0)，无论序列多长，目标值始终在固定范围内
- **单调**：越接近终止，R(t) 越靠近0；剩余越多，越靠近-1
- **Bellman一致**：满足 R(t) = r(t) + γR(t+1)，完全契合标准值函数框架

## 可扩展的价值预训练

### 训练目标

token级均方误差：

```
L = E[(V(s_t) - G_t)^2]
```

四大优势：无需标注、信号密集、无偏、可扩展。

### 三轴Scaling验证

| 轴 | 范围 | 结论 |
|----|------|------|
| 模型规模 | 0.5B → 32B | 更大的模型始终带来更低的验证损失 |
| 训练prompt数 | 10k → 100k | 更广泛的数据覆盖持续改善长度建模质量 |
| 每prompt采样数 | n=1 → n=16 | 更多completion轨迹带来更强的监督 |

## 三种推理时应用

### 应用一：精确长度控制

LIFEBench基准结果（问答、摘要、推理、创意写作，中英文各180条）：

| 模型 | 长度得分 | 长度偏差 |
|------|---------|---------|
| Qwen2.5-3B + LenVM (1.5B) | **62.6** | 56% |
| Qwen2.5-7B + LenVM | **64.8** | 44% |
| GPT-5.4 | 37.4 | - |
| Claude-Opus-4-6 | 35.5 | - |
| Gemini-3.1-Pro | 49.3 | - |
| Qwen2.5-3B 基线 | 25.6 | 83% |

3B开源模型全面击败GPT-5.4和Claude-Opus-4-6。

### 应用二：性能-效率连续权衡

GSM8K实验结果（token预算200）：

| 方法 | Pass@1 |
|------|--------|
| 硬截断基线 | ≈ 6% |
| LenVM引导解码 | **≈ 63%** |

相差**10倍**！揭示重要事实：基础模型本身就具备用更短路径解题的能力，只是通常选不到这些路径。

### 应用三：生成长度预测

32B模型平均相对误差（MRE）：数学9.8%、代码14.9%、指令跟随17.1%。

对推理系统的批处理分组、KV缓存预分配、请求优先级排序有直接价值。

## 额外发现：哪些token在"延长"或"收束"推理？

- **延长推理的token (δ > 0)**：wait, but, ah, think, consider（"ah"频繁出现在"Aha Moment"）
- **收束推理的token (δ < 0)**：therefore, clearly, perfect, ✅ 🎉

## 核心结论

- 将控制粒度从序列级推进到**token级**
- 证明了**生成长度是可扩展价值预训练的一个新维度**
- 为RL训练提供长度专属的价值基线，可用于PPO中作为密集优势信号
